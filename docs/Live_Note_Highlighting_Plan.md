# Live Note Highlighting Plan

## Overview

This document describes the design and implementation plan for **live note highlighting** in the browser IDE: as a compiled MML track plays back, the Monaco editor decorates the currently-sounding notes (and optionally their instrument definitions) in real time.

The feature has two complementary goals:

1. **Note tracking** — highlight the source token(s) of every note that is currently producing sound, with color-coding per part/channel.
2. **Instrument tracking** — optionally dim-highlight the instrument definition line that was last selected on each channel.

---

## Current State

### What already exists

| Component | Relevant code | What it does |
|-----------|---------------|-------------|
| `MonacoEditor.tsx` | `currentLineDecorations` | Highlights the current line with `trace-current-line` CSS class |
| `TraceService` | `timingMap: Map<number, Position>` | Maps playback time (ms) → `{line, column}` |
| `App.tsx` | `createTimingMap()` | Builds the timing map after a successful compile |
| `AudioService` | `parseVgmCommands()` | Parses VGM binary to `ParsedVgmCommand[]` with `timeSamples, chip, addr, data` |
| `TraceService` | `updateActiveParts()` | Rotates "active part" indicator — **simulated, not data-driven** |
| `types/index.ts` | `Position`, `TraceEvent` | Type definitions for positions and trace events |

### The critical gap

`createTimingMap` in `App.tsx` is a naive linear approximation — it divides total duration evenly across source lines:

```typescript
const msPerLine = durationMs / totalLines;
for (let line = 0; line < totalLines; line++) {
  timingMap.set(Math.round(line * msPerLine), { line: line + 1, column: 1 });
}
```

This ignores actual note positions and durations, produces line-level rather than token-level precision, and is completely wrong when lines have unequal numbers of notes. Accurate highlighting requires a **source map** generated by the Rust compiler itself.

### What does not yet exist

- Source-position annotations on AST nodes (`MmlNode` variants carry no `line`/`column`)
- A source map output from the code generator linking sample offsets to source tokens
- A WASM API to retrieve the source map from a compile result
- Per-note range decorations (column-precise, multi-note concurrency)
- Instrument line tracking

## Phase 1: Rust — Source Spans on AST Nodes

### 1.1 Span type

Add a `Span` type to `mml2vgm-rs/src/error.rs` (alongside `Position`):

```rust
/// A source span: start and end position in the source file
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Span {
    pub start: Position,
    pub end: Position,   // inclusive end (last character of token)
}
```

### 1.2 Spanned wrapper

Add a generic `Spanned<T>` wrapper, or add an optional span field to the specific AST nodes that matter:

```rust
// Option A — generic wrapper (clean, but verbose)
pub struct Spanned<T> {
    pub node: T,
    pub span: Span,
}

// Option B — add span field to each relevant node (targeted, minimal churn)
pub struct Note {
    /* existing fields … */
    pub span: Option<Span>,
}
pub struct Rest {
    /* existing fields … */
    pub span: Option<Span>,
}
pub struct InstrumentSelection {
    pub number: usize,
    pub span: Option<Span>,
}
```

**Option B is recommended** — it minimises changes to the parser and codegen, and the `Option` keeps backward compatibility.

### 1.3 Parser changes

The lexer already maintains `current_line` / `current_column` and exposes `current_position() -> Position`. The parser should capture `span_start` before consuming a note token and `span_end` after:

```rust
// parser.rs — inside parse_note_command()
let span_start = self.current_position();
let note_char = /* consume note token */;
// ... consume optional accidental, octave, duration, dot, tie ...
let span_end = self.current_position();
let note = Note { letter, accidental, octave, duration, dotted, tied,
                  volume: None, instrument: None,
                  span: Some(Span { start: span_start, end: span_end }) };
```

Apply the same pattern to `Rest` (for `r`/`R`) and `InstrumentSelection` (for `@NNN`).

## Phase 2: Rust — Source Map Emission in the Code Generator

### 2.1 NoteEvent type

Add to `mml2vgm-rs/src/compiler/codegen/`:

```rust
/// A single note event with timing and source location
#[derive(Debug, Clone, serde::Serialize)]
pub struct NoteEvent {
    /// Sample offset at note-on (44100 Hz)
    pub sample_start: u64,
    /// Sample offset at note-off (i.e. start + quantized duration)
    pub sample_end: u64,
    /// Part name (e.g. "A1", "Y01")
    pub part: String,
    /// MIDI note number 0–127
    pub note_midi: u8,
    /// Instrument number (from last `@NNN` in this part), or 0 if none
    pub instrument: u32,
    /// Source line (1-indexed)
    pub line: usize,
    /// Source column start (1-indexed)
    pub col_start: usize,
    /// Source column end (1-indexed, inclusive)
    pub col_end: usize,
}
```

### 2.2 SourceMap type

```rust
#[derive(Debug, Default, Clone, serde::Serialize)]
pub struct SourceMap {
    pub events: Vec<NoteEvent>,
}
```

### 2.3 Codegen changes

In `VgmCodegen` (or the generic `CodeGenerator` trait), add a `source_map: SourceMap` accumulator. When emitting a note-on event for a part:

```rust
// vgm.rs — inside the note emit block
if let Some(span) = note.span {
    self.source_map.events.push(NoteEvent {
        sample_start: *time,
        sample_end: *time + note_on_samples as u64,
        part: current_part_name.clone(),
        note_midi: note.midi_note(),
        instrument: state.current_instrument,
        line: span.start.line,
        col_start: span.start.column,
        col_end: span.end.column,
    });
}
```

### 2.4 CompileResult changes

`mml2vgm-rs/src/compiler/compiler.rs` — add `source_map` to the compile result:

```rust
pub struct CompileOutput {
    pub data: Vec<u8>,
    pub info: CompileInfo,
    pub source_map: SourceMap,   // NEW
}
```

## Phase 4: Browser IDE — Consume Source Map

### 4.1 TypeScript types (`types/index.ts`)

```typescript
/** A single note event from the compiler source map */
export interface SourceMapEvent {
  sample_start: number;   // samples at 44100 Hz
  sample_end: number;
  part: string;
  note_midi: number;      // 0–127
  instrument: number;
  line: number;           // 1-indexed
  col_start: number;      // 1-indexed
  col_end: number;        // inclusive
}

/** Full source map returned from compile */
export interface SourceMap {
  events: SourceMapEvent[];
  sampleRate: number;     // always 44100 for now
}
```

### 4.2 CompileStore / compile result

In `compileStore.ts` (or wherever compile results are stored), add:

```typescript
sourceMap: SourceMap | null;
```

Populate it in the worker result handler by parsing `result.source_map_json`.

### 4.3 TraceService update

Replace the naïve `createTimingMap` + `updatePosition` logic with interval-based lookup over `SourceMapEvent[]`:

```typescript
/** Returns all NoteEvents that are sounding at `timeSamples` */
private getActiveEvents(timeSamples: number): SourceMapEvent[] {
  if (!this._sourceMap) return [];
  return this._sourceMap.events.filter(
    e => timeSamples >= e.sample_start && timeSamples < e.sample_end
  );
}
```

For large source maps (thousands of events) a simple interval tree or sorted binary-search on `sample_start` would be more efficient, but a linear scan is fine for typical MML files (<5 000 events).

Emit multiple `onNoteEvents` to listeners per time update:

```typescript
export interface TraceEventListener {
  /* existing … */
  onNoteEvents?: (events: SourceMapEvent[], timeSamples: number) => void;
}
```

## Phase 5: Monaco Editor — Decorations

### 5.1 Per-note range decorations

`MonacoEditor.tsx` receives `activeNoteEvents: SourceMapEvent[]` from the parent (via props or context). On each animation frame update:

```typescript
// Clear previous note decorations
const newDecorations = activeNoteEvents.map(evt => ({
  range: new monaco.Range(evt.line, evt.col_start, evt.line, evt.col_end + 1),
  options: {
    inlineClassName: `note-playing note-part-${cssPartClass(evt.part)}`,
    hoverMessage: { value: `**${midiNoteToName(evt.note_midi)}** — part ${evt.part}` },
  },
}));
editor.deltaDecorations(prevNoteDecorations, newDecorations);
```

Add to `MonacoEditorProps`:

```typescript
activeNoteEvents?: SourceMapEvent[];   // notes currently sounding
```

### 5.2 CSS classes

In `index.css` (or a dedicated `trace.css`):

```css
/* Active note token highlight */
.note-playing {
  border-radius: 2px;
  opacity: 0.85;
}

/* Per-part colour palette (cycles modulo 8) */
.note-part-0 { background-color: rgba(97, 175, 239, 0.30); }
.note-part-1 { background-color: rgba(152, 195, 121, 0.30); }
.note-part-2 { background-color: rgba(229, 192, 123, 0.30); }
.note-part-3 { background-color: rgba(198, 120, 221, 0.30); }
.note-part-4 { background-color: rgba(86, 182, 194, 0.30); }
.note-part-5 { background-color: rgba(224, 108,  117, 0.30); }
.note-part-6 { background-color: rgba(209, 154,  102, 0.30); }
.note-part-7 { background-color: rgba(130, 170,  255, 0.30); }
```

A mapping function converts a part name (e.g. `"A1"`, `"Y03"`) to a stable 0–7 index (e.g. `hashCode(partName) % 8`).

### 5.3 Whole-line gutter marker (optional)

Keep the existing `trace-current-line` whole-line decoration but demote it to a faint left-gutter marker so it doesn't compete with the inline token highlights.

### 5.4 Instrument line tracking (optional / Phase 6)

When a part changes its instrument (`@NNN`), record the line of that `InstrumentSelection` node in the source map (add `instrument_line` to `NoteEvent`, or emit a separate `InstrumentEvent`). Display a gutter icon on that line while the instrument is selected.

## Phase 6: Visual Polish

### 6.1 Note name tooltip / status bar

When at least one note event is active, show a transient status-bar badge:

```
♩ C4 · E4 · G4   (part A1)
```

Reuse the existing `StatusBar.tsx` component; add a `playingNotes: string` prop.

### 6.2 Smooth transition

Instead of snapping decorations on every audio tick, use a short CSS transition (`transition: background-color 60ms linear`) on `note-playing` so fast tempos don't produce a distracting flash.

### 6.3 "Playhead" gutter icon

Add a narrow vertical bar (SVG glyph margin decoration) on the leftmost active line to act as a playhead indicator that is visible even when the editor is scrolled.

## Alternatives Considered

### Alternative A: VGM register-write heuristic (no Rust changes)

The `AudioService` already parses VGM commands and knows which register writes occur at which sample offsets. It's theoretically possible to reverse-engineer note frequency from OPM/OPL key registers to infer which note is playing, then do a fuzzy text search to find the matching source token.

**Rejected**: extremely brittle across chip families, impossible for PSG noise channels, and still produces no column precision.

### Alternative B: Line-level source map only (simpler Rust changes)

Instead of tracking per-token column spans, only record `(sample_start, line, part)` per note event. This is easier to implement (no `Span` needed, just `Position` at note parse time) and still achieves line-level highlighting identical to the current `trace-current-line` decoration, but driven by real timing data.

**Viable as a first step** — can ship line-level accuracy first, then add column precision in a follow-up.

### Alternative C: Sidecar JSON file

The compiler could write a `.map` JSON file alongside the `.vgm` output. The browser IDE would load it as a second fetch.

**Rejected for browser IDE**: adds a round-trip and complicates the WASM API. Fine for the CLI / egui app.

## Open Questions

1. **Multi-file / include**: When `#include` is used, source positions refer to the included file. The source map should carry a `file_index` field and the compile result should return the ordered list of included filenames. (Out of scope for initial implementation.)

2. **Loop unrolling**: A `[abc]3` loop emits the same notes three times with different sample offsets but the same source positions. All three repetitions can share the same source token decoration — the editor just highlights it three times in sequence. No special handling needed.

3. **Sample rate variability**: The `NoteEvent.sample_start` is in 44100 Hz samples. If the audio context is resampled (unlikely in modern browsers), the conversion `ms = sample / 44100 * 1000` may drift slightly. Cache the sample rate at compile time.

4. **egui desktop app**: The same source map could be used in the egui app's `Editor` panel (`editor.rs`) via the Rust `SourceMap` type directly, without JSON serialisation. Out of scope here but the data model is compatible.

5. **Performance**: For very long tracks (>10 minutes) or dense arrangements (>10 parts), the source map may contain tens of thousands of events. A flat Vec scan per animation frame (~60 Hz) over 50 000 events costs ~3 ms per frame — acceptable, but an interval tree would reduce this to <0.1 ms if needed.
